Trust Center

Cryptographic evidence infrastructure for AI decision records.

Obrioxia provides tamper-evident audit infrastructure. Every decision record is hash-chained, encrypted, and anchored. The system detects modification — it does not prevent it. Technical details available on request.

Encryption Standards

All data in transit is protected with TLS. Data at rest is protected using per-event envelope encryption: AES-256-GCM with a unique Data Encryption Key (DEK) per record, wrapped with RSA-OAEP using the writer's public key. The production service holds only the public key — it can seal records but never decrypt historical data.

See Technical Details

Tamper-Evident Cryptographic Chaining

Every decision event is cryptographically chained to its predecessor using SHA-256 hashing. Modifications are detectable through verification, subject to the anchoring interval. The chain is linear, rooted at a genesis block, with uniqueness enforced via indexed constraints on each link.

See Technical Details

Privacy & Per-Record Data Destruction

Obrioxia resolves the conflict between tamper-evident logs and the Right to be Forgotten. By encrypting PII with write-only per-event keys before hashing, individual records can be cryptographically destroyed. When a record's key material is permanently nulled, PII becomes computationally irrecoverable, while the audit chain remains intact and verifiable.

Data Residency & Retention

Obrioxia manages data retention through cryptographic access revocation. Retention periods are enforced by permanently destroying the encryption key for expired records. Encrypted data remains in the tamper-evident chain but becomes computationally irrecoverable.

Anchoring & External Verification

Admin-triggered cryptographic checkpoints create deterministic SHA-256 digests over ranges of chain entries. Each checkpoint chains to the previous via previous_checkpoint_hash. Local anchor artifacts are written to disk for offline archival.

  • ►Optional: RFC 3161 TSA timestamping when TSA_URL is configured.
  • ►Optional: S3-compatible witness sink when WITNESS_S3_ENABLED=1 is set.

Limitations

Obrioxia is tamper-evident, not tamper-proof. An operator with infrastructure access can modify data; the system detects such modifications. Event completeness cannot be guaranteed — if an event is never submitted, the system cannot detect its absence. Technical details available on request.

Checkpoint Verification

Obrioxia provides admin-gated endpoints for checkpoint management and retrieval:

  • ►POST /api/admin/checkpoints — create a checkpoint over a range of chain entries
  • ►GET /api/admin/checkpoints/latest — retrieve the latest checkpoint hash and metadata
  • ►GET /api/admin/witness/latest — retrieve latest witness details and presigned URL (when S3 is enabled)

Architecture Limitations

Current production architecture constraints — included for transparency:

  • —Single MongoDB instance (no multi-region replication)
  • —Linear hash chain (no Merkle tree)
  • —Tamper-evident, not tamper-proof — an operator with infrastructure access can modify data; the system detects such modifications
  • —No public transparency log
  • —Checkpoints are admin-triggered, not automatic
  • —Key ceremony generates keys but does not auto-rotate the active encryptor
